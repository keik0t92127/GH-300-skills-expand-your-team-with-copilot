# OctoAcme プロジェクト管理ドキュメント

## このフォルダの目的

このフォルダ（`docs/`）は、OctoAcme が使用するプログラムプロセス文書を集約し、検索可能でバージョン管理された形式で保存するためのものです。主な目的は以下のとおりです。

- プロジェクト管理プロセスを単一の情報源（GitHub リポジトリ）に集約する
- チームメンバーが一貫したプロセスを参照・遵守できるようにする
- プロセスの変更をバージョン管理し、変更履歴を追跡できるようにする
- 新しいメンバーのオンボーディングを支援する

## OctoAcme プロジェクト管理プロセス概要

### ライフサイクルと基本ワークフロー

OctoAcme のプロジェクトは「Initiation（開始）→ Planning（計画）→ Execution（実行）→ Release（リリース）→ Retrospective（振り返り）」という5段階のライフサイクルに沿って進められます。開始フェーズではプロジェクトチャーターを作成してステークホルダーを特定し、GitHub 上にリポジトリとプロジェクトボードを初期化します（[プロジェクト開始](octoacme-project-initiation.md)）。計画フェーズでは Product Manager がバックログを精緻化し、すべてのストーリーに受け入れ基準を定義した後、Developer と QA と合意の上でスプリント計画を立てます（[プロジェクト計画](octoacme-project-planning.md)）。実行フェーズでは Developer が小さく集中したプルリクエストを提出し、CI チェック（ユニットテスト・統合テスト・静的解析・セキュリティスキャン）が全てグリーンになってからレビューを依頼します（[実行とトラッキング](octoacme-execution-and-tracking.md)）。プロジェクトボードのカラム（Backlog → Ready → In Progress → In Review → Done）が作業状態の唯一の可視化手段となり、マージにはレビュアー最低1名の承認が必要です。

### 役割とコミュニケーション戦略

プロジェクトには PM・PdM・Dev・QA・Stakeholders の5つの主要な役割が参加します。PM がプロジェクト全体のデリバリーを所有し、PdM がプロダクトビジョンとバックログを管理します（[役割とペルソナ](octoacme-roles-and-personas.md)）。コミュニケーションは一定のリズムで運営されており、毎日の非同期スタンドアップ、週次の全チーム同期ミーティング、スプリント終了時のデモ・レビューセッション、週次のステークホルダー向け書面更新という4つのコアイベントが定義されています。すべての意思決定と状態情報は GitHub リポジトリを唯一の情報源（Single Source of Truth）として管理します。リスクはリスクレジスターに記録され、週次同期でレビューされ、高リスク・高影響度の項目はステークホルダーにエスカレーションされます（[リスクとコミュニケーション](octoacme-risks-and-communication.md)）。

### 品質保証とリリース

品質は開発プロセス全体に組み込まれています。受け入れ基準はすべてのストーリーに必須で、CI パイプラインは PR ごとに自動実行されます。CI が失敗している PR はマージできません。リリースは事前の go/no-go 基準（全 CI グリーン・QA サインオフ・セキュリティスキャン完了・リリースチェックリスト完了・ステークホルダー承認）をすべて満たした場合のみ実施されます。また、すべてのリリースにはロールバック計画が必要で、ステージング環境でテスト済みであることが条件です（[リリースとデプロイ](octoacme-release-and-deployment.md)）。リリース後は本番環境を24時間モニタリングし、エラーレートやパフォーマンス指標を監視します。

### 振り返りと継続的改善

OctoAcme はすべてのスプリント終了後とリリース後に振り返りを実施します。振り返りでは「うまくいったこと・うまくいかなかったこと・学んだこと」を整理し、アクションアイテムを GitHub Issue として記録します。プロセス改善が合意された場合は同スプリント内でドキュメントを更新します。PM は月次でサイクルタイムや不具合率などのメトリクスをレビューし、チームへのフィードバックループを維持します（[振り返りと継続的改善](octoacme-retrospective-and-continuous-improvement.md)）。これらの取り組みにより、チームは繰り返しのミスを防ぎ、デリバリー品質を継続的に向上させます。

## 推奨される読み進め方

プロセスを初めて学ぶ方は、以下の順で読み進めることをお勧めします。

1. **[プロジェクト管理概要](octoacme-project-management-overview.md)** — ライフサイクル全体と各フェーズの関係を把握する
2. **[役割とペルソナ](octoacme-roles-and-personas.md)** — 誰が何を担当するかを理解する
3. **[プロジェクト開始](octoacme-project-initiation.md)** — プロジェクトのスタート方法を学ぶ
4. **[プロジェクト計画](octoacme-project-planning.md)** — バックログ精緻化とスプリント計画を理解する
5. **[実行とトラッキング](octoacme-execution-and-tracking.md)** — 日々の開発ワークフローを確認する
6. **[リスクとコミュニケーション](octoacme-risks-and-communication.md)** — リスク管理とステークホルダー連絡の方法を学ぶ
7. **[リリースとデプロイ](octoacme-release-and-deployment.md)** — リリース基準と手順を確認する
8. **[振り返りと継続的改善](octoacme-retrospective-and-continuous-improvement.md)** — 改善サイクルの仕組みを理解する
